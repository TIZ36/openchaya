// promhttp.Handler() 返回一个现成的 http.Handler,
// 它会把「默认注册表」里的所有指标渲染成文本。
http.Handle("/metrics", promhttp.Handler())
http.ListenAndServe(":8080", nil)

跑起来：

go run .
# 另开一个终端
curl localhost:8080/metrics

你会看到一堆 go_ 和 process_ 开头的指标(GC 次数、内存、协程数……)。这些是库白送的,你一行没写就有了。

到这里,你已经有一个能被 Prometheus 抓取的程序了。神奇的地方就一句 promhttp.Handler() 。先消化这一步,再往下。

第 2 步:认识四种指标类型(还是先不写代码)

Prometheus 只有 4 种基本类型。先搞懂"什么时候用哪个",写代码就不会选错:

类型一句话典型场景能不能减小
Counter(计数器)只增不减的累加值请求总数、错误总数❌ 只能增
Gauge(仪表盘)可上可下的瞬时值当前内存、在线人数✅ 可增可减
Histogram(直方图)把观测值分到若干「桶」里统计分布请求延迟、响应大小——
Summary(摘要)类似 Histogram,客户端算分位数延迟分位(少用)——

99% 的场景里你只会用到前三种,Summary 基本可以忘掉(它分位数不能跨实例聚合,大多数人用 Histogram 代替)。

选型口诀:

「累计发生了多少次」 → Counter
「此刻是多少」       → Gauge
「分布/快慢/大小」   → Histogram

我们下面就按 Counter → Gauge → Histogram 的顺序,各写一个。先从最简单的 Counter 开始——它就是一个只会变大的数字,适合"请求总数""错误总数"这类累计量。

记住一个原则:指标的命名要能自解释。http_requests_total 一眼就知道是"HTTP 请求的累计总数",后缀 _total 是 Counter 的社区约定。Gauge 不加后缀,Histogram 会自动生成 _bucket / _sum / _count 三组序列。

另外别忘了 label 的基数(cardinality)问题:label 取值组合越多,时间序列爆炸得越快。把 user_id、request_id 这种高基数维度塞进 label 是新手最常踩的坑,能直接把 Prometheus 内存撑爆。
